# Agent 团队改进方案（2026-04-07）

## 1. 目标
- 将当前团队表现从“单体强、协作一般”提升到“单体强、协作稳、闭环清晰”。
- 重点修复 3 个问题：
  1. 缺少统一验收基线
  2. 缺少任务状态流转机制
  3. manager 收口与心跳隔离不足

## 2. 本次测试结论
- 总体评价：7.5/10
- 单体产出能力：8.5/10
- 协作闭环能力：6/10
- 主要问题不在产出能力，而在调度、状态管理、交付闭环。

## 3. 成员改进项

### researcher
- 需求卡除了“规则/假设”外，必须补：
  - 2 条正例
  - 2 条易混淆样例
  - 1 组负面约束（明确“不做什么”）
- 输出优先级：字段定义 > 验收样例 > 风险点。

### coder
- 交付前必须执行“需求卡对照自检”：
  1. 输出字段是否完全一致
  2. 映射规则是否全覆盖
  3. 示例输入输出是否严格一致
  4. 边界条件是否和需求卡冲突
- 首次提交必须附“已核对项”摘要，不允许只给代码不说明规则对齐结果。

### office
- 提前介入，而不是只在最后润色。
- 需要在任务中段提供：
  - 用户视角验收清单
  - 使用说明草稿
  - 适用边界与风险提示
- 目标是帮助 manager 更早判断“是否已达到可交付标准”。

### manager
- 重点改进项：
  1. 明确任务状态迁移
  2. 交付完成后立即结项
  3. 心跳巡查与真实业务回复分离
  4. 用固定模板做收口，不靠临场判断

## 4. 统一流程改进

### 4.1 任务状态流转
统一使用以下状态：
- 待拆解
- 进行中
- 待审核
- 待交付
- 已完成
- 已关闭

### 4.2 进入“已完成”的条件
必须同时满足：
- researcher 需求口径已锁定
- coder 产物通过规则核对
- office 对外交付说明已就绪
- manager 已完成审核并确认可交付

### 4.3 高频巡查退出条件
满足以下任一条件，应从“活跃任务”移除：
- 已明确结项
- 已交付用户并无待修订项
- 任务转为“已关闭”

## 5. 固定模板

### 5.1 researcher 需求卡最小模板
- 任务理解
- 输入定义
- 输出定义
- 规则/假设
- 风险点
- 示例输入（含正例/易混淆样例）

### 5.2 coder 自检模板
- 字段一致：是/否
- 规则一致：是/否
- 示例一致：是/否
- 边界检查：是/否
- 已知偏差：无/列出

### 5.3 manager 收口模板
- 当前版本：谁已定版，谁待修
- 剩余差异：最多 3 条
- 下一步负责人：唯一责任人
- 完成判定：满足什么条件即可交付
