# Agent 团队标准操作流程（SOP）

## 1. 适用范围
适用于 manager / researcher / coder / office 参与的多 Agent 协作任务，目标是减少规则漂移、状态漂移和交付闭环不清的问题。

## 2. 角色分工

### manager
负责：
- 接收需求
- 拆解任务
- 指定验收标准
- 跟进进度
- 审核成果
- 统一交付
- 任务结项

### researcher
负责：
- 明确任务理解
- 定义输入/输出
- 提供规则/假设
- 给出样例与边界
- 输出风险点

### coder
负责：
- 产出可运行实现
- 对齐 researcher 规则
- 完成自检
- 提供使用说明与示例

### office
负责：
- 从用户视角整理成果
- 输出交付说明
- 补充验收清单
- 提示适用边界与风险

## 3. 统一状态流转
所有复杂任务只允许使用以下状态：
- `待拆解`
- `进行中`
- `待审核`
- `待交付`
- `已完成`
- `已关闭`

### 状态定义
- **待拆解**：刚收到任务，尚未形成分工
- **进行中**：子任务已下发，仍在执行
- **待审核**：已有正式产物，等待 manager 审核
- **待交付**：已审核通过，等待统一发给用户
- **已完成**：已交付且无待修订项
- **已关闭**：任务终止、取消或不再继续

## 4. 标准执行流程

### 步骤 1：任务接收
manager 判断任务类型：
- 简单查询：可直接处理
- 复杂协作：进入拆解流程

### 步骤 2：任务拆解
manager 必须一次性明确：
- 目标产物
- 参与角色
- 每个角色交付内容
- 验收标准
- 当前任务状态

### 步骤 3：研究员先锁规则
researcher 优先输出需求卡，至少包含：
- 任务理解
- 输入定义
- 输出定义
- 规则/假设
- 风险点
- 正例样本
- 易混淆样本
- 负面约束（明确不做什么）

### 步骤 4：程序员实现并自检
coder 在收到规则后开始实现，交付时必须附自检结果：
- 字段一致：是/否
- 规则一致：是/否
- 示例一致：是/否
- 边界检查：是/否
- 已知偏差：无/列出

### 步骤 5：办公专家补用户视角交付说明
office 不应只在最后润色，应尽早补充：
- 使用说明
- 用户视角验收清单
- 适用范围
- 风险提示

### 步骤 6：manager 收口审核
manager 统一审核时使用固定结构：
1. 当前版本：谁已定版，谁待修
2. 剩余差异：最多 3 条
3. 下一步负责人：唯一责任人
4. 完成判定：满足什么条件即可交付

### 步骤 7：交付与结项
交付用户后，manager 必须立即：
- 将任务状态改为 `已完成` 或 `已关闭`
- 更新 `MEMORY.md`
- 记录完成时间
- 记录交付物路径
- 标记是否仍有遗留问题

## 5. 验收基线

### researcher 验收要求
- 规则清晰
- 输入输出明确
- 边界条件明确
- 至少有正例与易混淆样例

### coder 验收要求
- 可运行
- 与规则逐项一致
- 示例输入输出一致
- 无明显未说明偏差

### office 验收要求
- 用户能看懂
- 使用方式明确
- 适用边界明确
- 风险提示明确

### manager 验收要求
- 审核结论明确
- 状态更新及时
- 交付闭环完整
- 不留下“实际完成但系统仍显示进行中”的漂移

## 6. 心跳处理规则
- 心跳只看当前 workspace 可证实事实
- 不从 prior chats 推断任务仍在执行
- 若 `MEMORY.md` 仍挂活跃任务，但实际已交付，应优先提醒“立即结项”
- 只有在无待处理任务、无真实活跃任务、无异常标记、无超时审核事项时，才回复 `HEARTBEAT_OK`

## 7. 常见失败模式与纠正动作

### 失败模式 1：规则漂移
现象：coder 首版实现与 researcher 规则不一致。
纠正：强制执行“需求卡对照自检”。

### 失败模式 2：状态漂移
现象：任务实际上已完成，但 `MEMORY.md` 仍是进行中。
纠正：交付后立即结项，并纳入心跳检查项。

### 失败模式 3：office 过晚介入
现象：直到最后才补说明，无法帮助提前判断是否可交付。
纠正：在任务中段就输出验收清单与使用说明草稿。

### 失败模式 4：manager 收口过慢
现象：多轮往返，已定版内容没有及时宣布定版。
纠正：统一使用收口模板，每轮只保留最多 3 条差异。

## 8. 最小收口模板

### 对内
- 当前版本：
- 剩余差异：
- 下一步负责人：
- 完成判定：

### 对用户
- 当前进展：
- 已完成部分：
- 剩余事项：
- 下一步动作：

## 9. 复盘要求
每次多 Agent 协作结束后，至少复盘：
- 哪个角色最稳
- 哪个环节最慢
- 是否出现规则漂移
- 是否出现状态漂移
- 下次优先改什么

## 10. 当前版本结论
当前团队最需要长期坚持的不是增加新角色，而是：
1. 先锁规则
2. 再做实现
3. 及时收口
4. 立即结项
