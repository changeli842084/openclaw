# MULTI_AGENT_COLLAB_SOP.md

## 目标
用于规范 manager / researcher / coder / office 之间的任务交接，减少长提示漂移、输出格式不统一、回收验收困难的问题。

## 一、统一任务信封
每次下发任务，尽量包含以下字段：

- `task_id`: 唯一任务编号
- `role`: 当前执行角色
- `goal`: 本轮唯一目标
- `inputs`: 上游输入摘要
- `deliverable`: 本轮交付物要求
- `format`: 输出格式要求
- `done_when`: 完成判定标准
- `next_agent`: 下一棒是谁
- `constraints`: 约束（语言、长度、是否允许代码/联网等）

## 二、推荐交接格式

### 1. 研究员 → 程序员
固定输出：
1. 任务理解
2. 输入定义
3. 输出定义
4. 规则/假设
5. 风险点
6. 示例输入

### 2. 程序员 → 办公专家
固定输出：
1. 实现说明
2. 使用方式
3. 示例输入
4. 示例输出
5. 已知局限
6. 需业务方复核项

### 3. 办公专家 → 用户
固定输出：
1. 工具用途
2. 输入说明
3. 输出字段解释
4. 使用流程
5. 人工复核建议

## 三、最小验收规则
manager 回收结果时，至少检查：

- 是否严格按约定结构输出
- 是否能被下一棒直接消费
- 是否有缺失字段或含糊表达
- 是否有不必要寒暄、跑题内容
- 是否给出风险/边界说明

## 四、联调建议

### 推荐做法
- 优先使用短任务链，避免超长自由文本提示
- 优先要求结构化输出（JSON / 固定章节）
- 每一棒只做一件事，不混合“分析 + 实现 + 交付”
- 对下游说明要显式，不依赖隐含上下文

### 不推荐做法
- 一次提示里塞入过多背景和多重目标
- 不指定输出格式，指望 Agent 自行理解
- 不设验收标准就直接交棒
- 用主会话历史当作唯一事实源

## 五、推荐的最小联调模板

```text
任务编号：<task_id>
当前角色：<researcher|coder|office>
本轮目标：<一句话>
上游输入：<3~6条>
交付要求：<明确列点>
输出格式：<JSON 或固定章节>
完成标准：<2~4条>
下一棒：<next_agent>
额外约束：<语言/长度/是否写代码>
```

## 六、当前暴露出的主要问题
1. 长提示下输出容易漂移，导致 manager 回收困难。
2. 缺少统一 task_id / deliverable / next_agent 字段，接力成本高。
3. 不同 Agent 输出风格差异大，验收口径不统一。
4. 历史消息较多时，主会话结果不够“干净”，联调建议优先使用短链路和结构化格式。

## 七、后续执行原则
后续 manager 发起多 Agent 协作时，默认优先采用本 SOP。若任务特殊，再做定制化扩展。
